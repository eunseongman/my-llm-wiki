# DB 실행계획 감각을 잡기 위해 오늘 공부한 것

## 오늘 무엇을 공부했나

오늘 정리한 내용의 중심은 개별 용어 암기보다, 백엔드에서 성능을 볼 때 무엇을 중심축으로 잡아야 하는지에 가까웠다. JPA와 인덱스의 관계에서 출발해, 대량 업서트가 왜 빠른지, 배치 UPDATE와 staging table + merge는 언제 쓰는지, 문자열 콜레이션이 실제로 무엇에 영향을 주는지, 그리고 인덱스 컬럼에 함수를 걸면 왜 인덱스를 잘 못 타는지를 하나의 흐름으로 묶어 봤다.

처음에는 "JPA로도 인덱스를 유도할 수 있나"라는 질문에서 시작했다. 여기서 가장 중요한 정리는 JPA가 인덱스를 직접 쓰게 만드는 도구는 아니라는 점이었다. JPA를 써도 결국 실행되는 것은 SQL이고, 인덱스를 타느냐 아니냐는 DB가 실제로 받은 SQL 모양과 실행계획에 달려 있다. 즉, ORM은 편한 추상화이지만 성능 판단의 마지막 기준은 여전히 SQL과 EXPLAIN이다.

## 한 번에 이해할 흐름

이 흐름을 이해하려면 네 단계를 따라가면 된다.

첫째, JPA와 인덱스의 관계를 이해해야 한다. 루스 인덱스 스캔 같은 최적화는 단순히 인덱스를 만들었다고 생기지 않는다. `GROUP BY`, `MIN/MAX`, 인덱스 선두 컬럼 순서, 조회 컬럼 수처럼 SQL 형태가 맞아야 한다. 그래서 백엔드에서는 JPA 코드를 예쁘게 쓰는 것보다 DB가 인덱스를 타기 좋은 SQL을 생성하게 만드는 것이 중요하다.

둘째, 대량 데이터 반영 전략을 구분해야 한다. 업서트는 신규와 기존 데이터가 섞여 있을 때 애플리케이션이 일일이 존재 여부를 조회하지 않고 DB가 유니크 인덱스 기준으로 한 번에 처리하게 만들어 준다. 이 방식은 네트워크 왕복, 인덱스 탐색, SQL 실행 오버헤드를 줄이는 데 강하다. 다만 대부분이 기존 데이터 수정이라면 배치 UPDATE가 더 적합할 수 있고, 수십만 건 이상을 안정적으로 반영하면서 검증과 재처리까지 챙겨야 하면 staging table + merge 전략이 더 유리하다.

셋째, 콜레이션은 문자열 정렬 옵션 정도가 아니라 문자열 비교 규칙이라는 점을 이해해야 한다. 문자셋이 저장 가능한 글자 범위를 정한다면, 콜레이션은 그 글자들을 어떤 기준으로 같다고 보고 어떤 순서로 비교할지를 정한다. 그래서 대소문자 구분, 정렬 순서, `UNIQUE INDEX` 충돌 여부까지 영향을 준다.

넷째, 인덱스 컬럼에 함수를 걸면 왜 인덱스가 불리해지는지도 연결된다. `DATE(created_at)`이나 `UPPER(name)`처럼 컬럼 값을 가공하면 DB는 원래 인덱스가 정렬해 둔 값을 그대로 쓰기 어렵다. 그래서 가능하면 컬럼 원형을 유지한 비교나 범위 조건으로 바꾸는 것이 유리하다.

## 핵심 개념 정리

오늘 같이 묶인 핵심은 다섯 가지다.

첫 번째는 "ORM을 써도 성능 판단은 SQL과 실행계획이 한다"는 점이다. JPA는 쿼리를 표현하는 도구이지, DB 최적화를 자동으로 보장해 주는 도구는 아니다.

두 번째는 업서트의 장점과 한계다. 업서트는 존재 확인과 분기 로직을 애플리케이션 밖으로 밀어내 DB 내부에서 유니크 인덱스를 기준으로 처리하게 해 주기 때문에 강력하다. 하지만 충돌이 너무 많거나 사실상 대부분이 수정이라면 오히려 부담이 커질 수 있다.

세 번째는 배치 UPDATE와 staging table + merge의 역할 분리다. 배치 UPDATE는 이미 존재하는 row를 많이 고칠 때 단순하고 효율적이다. staging table + merge는 대량 적재, 검증, 중복 제거, 재처리 같은 운영 관점까지 함께 챙길 때 유리하다.

네 번째는 콜레이션이다. 문자열 비교 규칙이므로 조회 결과, 정렬, 그룹화, 유니크 제약에 모두 연결된다.

다섯 번째는 함수 적용과 인덱스의 관계다. 인덱스는 원래 저장된 값 기준으로 정렬되므로, `함수(컬럼)` 형태의 조건은 대체로 인덱스를 직접 활용하기 어렵게 만든다.

## 내가 다시 볼 포인트

다시 읽을 때 가장 먼저 붙잡아야 할 문장은 이것이다. "백엔드 성능은 프레임워크가 아니라 DB가 실제로 받은 SQL과 실행계획이 결정한다." 오늘 이야기한 JPA, 업서트, 콜레이션, 함수 적용 문제는 전부 이 문장 아래에서 다시 연결된다.

다음에 이어서 공부하면 좋은 주제도 자연스럽다. 첫 번째는 `EXPLAIN`을 읽는 법이다. 지금 정리한 개념들은 실제 실행계획을 보면 훨씬 선명해진다. 두 번째는 MySQL에서 인덱스를 잘 타는 조건식과 못 타는 조건식을 예시로 비교하는 것이다. 세 번째는 JPA에서 성능 민감 구간을 Querydsl과 native query로 어떻게 분리할지 보는 것이다.
