# Task statement

Pokemon Showdown에서 상대 엔트리와 배틀 상태를 바탕으로 선출 및 턴별 행동을 추천하고, 그 근거와 리스크를 설명하는 프로젝트 아이디어를 깊게 인터뷰해 검증한다.

# Desired outcome

- 프로젝트의 진짜 목표를 분명히 한다.
- 프로젝트 정체성이 RAG 중심인지 의사결정 엔진 중심인지 구분한다.
- 사용자 가치와 팀 프로젝트 적합성을 검증한다.
- MVP 범위를 좁힌다.

# Stated solution

- 배틀 시작 전 상대 엔트리 기반 선출 추천
- 배틀 중 상태 기반 행동 추천
- 추천 이유와 리스크 설명
- 메타 정보는 RAG 또는 검색 기반으로 보조
- 실제 플레이는 Pokemon Showdown에서 진행

# Probable intent hypothesis

사용자는 단순 정보 검색기가 아니라, 실전 배틀 상황에서 설명 가능한 의사결정 보조 시스템을 만들고 싶어 한다. 다만 현재 아이디어는 검색, 메타 요약, 상태 추론, 정책 추천, UX 입력 흐름이 혼재되어 있어 정체성과 MVP가 불명확하다.

# Known facts / evidence

- 사용자는 날카로운 검증을 원하며 칭찬을 금지했다.
- 인터뷰 목적은 목표 명확화, 정체성 구분, 범위 축소, 사용자 가치 검증, MVP 축소다.
- 확인해야 할 핵심 축으로 사용자 정의, 문제 정의, 프로젝트 정체성, 입력 방식, 추천 로직, 데이터 소스, MVP 범위, 팀 프로젝트 적합성을 제시했다.

# Constraints

- 지금 단계에서는 구현이 아니라 인터뷰와 요구사항 정제에 집중한다.
- 질문은 표면 기능이 아니라 본질을 파고들어야 한다.
- 한 번에 5~8개씩 묶어서 질문해야 한다.
- 사용자의 답변에 대해 허점, 모순, 애매함을 후속 라운드에서 집요하게 압박해야 한다.

# Unknowns / open questions

- 실제 핵심 사용자는 누구인가.
- 가장 고통스러운 지점이 선출, 턴 의사결정, 복기 중 무엇인가.
- 사용자가 원하는 것이 정답, 설명, 학습, 승률 향상 중 무엇인가.
- 실시간 입력 UX가 가능한가.
- 추천 로직이 규칙 기반, 시뮬레이션 기반, 검색 기반 중 어디에 무게를 두는가.
- 메타/세트 데이터의 출처와 신뢰성은 무엇인가.
- 팀 프로젝트로서 역할 분담과 데모 설득력이 충분한가.

# Decision-boundary unknowns

- 어디까지 자동화하고 어디서 사용자 판단을 남길지
- MVP에서 무엇을 과감히 버릴지
- 실시간 실전 보조 vs 사후 복기/학습 도구 중 무엇을 우선할지
