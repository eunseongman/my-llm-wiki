## 작업 정의

RAG 팀 프로젝트의 주제를 기존 방향에서 수정하고, 강사 피드백을 받을 수 있을 정도로
구체적인 후보 주제와 문제정의를 만드는 인터뷰를 진행한다.

## 원하는 결과

- 후보 주제 3개 내외가 비교 가능한 수준으로 정리된다.
- 각 후보는 데이터 소스, 사용자 가치, 구현 범위, 차별점, 데모 시나리오까지 설명 가능해야 한다.
- 최종적으로는 강사님께 컨펌 받을 수 있는 주제 제안서 초안으로 이어질 수 있어야 한다.

## 사용자가 현재 말한 해결 방식

- 기존의 "신기술 인사이트 탐색" 성격의 주제를 접고 싶다.
- 대신 RAG 파이프라인 자체를 잘 보여줄 수 있는 주제로 바꾸고 싶다.
- 자격증 취득 도우미 같은 방향을 예시로 두고 있다.

## 의도 가설

- 댓글 축적이나 거대 지식베이스가 필요한 문제를 피하고 싶다.
- 결과물 자체보다 데이터 수집, 청킹, 임베딩, 저장, 검색, 리랭킹, 평가 최적화의
  설계 과정을 설득력 있게 보여주고 싶다.
- 공개 문서와 명확한 평가 기준이 있는 주제를 선택해 프로젝트 리스크를 낮추고 싶다.

## 확인된 사실 / 근거

- 회의에서 기존 주제는 "신기술"이라고 보기 어렵고, 댓글 축적 시간과 인사이트 품질이
  불확실하다는 문제가 제기되었다.
- 팀은 간단한 RAG 프로젝트라도 문서 시스템과 파이프라인 설계에 공을 들이는 방향을
  더 중요하게 보고 있다.
- 응급처치 가이드 사례처럼 문서 구조와 청킹 품질에 집중한 전례가 긍정 사례로 언급되었다.
- 익일까지 차기 주제 후보 3개를 구체화해 강사님께 컨펌받는 일정이 있다.
- "자격증 취득 도우미"와 "국가기술자격 출제기준 PDF"는 현재 떠오른 후보 방향이다.

## 제약 조건

- 익일까지 빠르게 설명 가능한 수준으로 정리되어야 한다.
- 신기술 담론이나 커뮤니티 댓글 축적에 의존하는 주제는 피하는 편이 낫다.
- 데이터는 공개적으로 구할 수 있고, 범위가 비교적 명확해야 한다.
- 팀 프로젝트이므로 구현 범위가 과도하지 않아야 한다.

## 아직 모르는 것

- 팀이 이 프로젝트에서 가장 보여주고 싶은 1순위 가치가 무엇인지
- 교수자 피드백에서 특히 중요하게 보는 평가 기준이 무엇인지
- 사용 대상이 실제 수험생인지, 시연용 가상 사용자 중심인지
- 이번 프로젝트에서 어디까지 만들고 어디까지는 일부러 제외할지
- 후보 3개를 모두 같은 축으로 비교할지, 한 방향만 깊게 밀지

## 의사결정 경계의 미정 항목

- 내가 독자적으로 정해도 되는 수준: 후보 구조, 문서 포맷, 비교 축
- 사용자 확인이 필요한 수준: 최우선 평가 기준, 비포함 범위, 1차 타깃 사용자
