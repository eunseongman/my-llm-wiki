## 작업 정의

LLM과의 학습 대화를 세션 안에 흩어지게 두지 않고, 시간이 지나도 다시 읽고 복습할 수 있는 지속적인 지식 산출물로 바꿔 주는 개인용 LLM 위키/블로그 시스템을 만든다.

## 원하는 결과

다음 조건을 만족하는 로컬 우선 지식 베이스의 MVP 계획을 정의한다.
- 사용자는 LLM과 대화하며 공부할 수 있다
- 그 대화에서 나온 핵심 학습 내용이 자동으로 마크다운 노트로 정리된다
- 그 노트들을 나중에 위키 형태로 다시 읽고, 필요하면 블로그형 요약 글로도 볼 수 있다
- 시스템은 출처를 보존하고 시간이 지날수록 점진적으로 내용을 다듬고 축적한다

## 확인된 사실 / 근거

- 현재 저장소는 `.omx/` 기본 구조 외에는 사실상 비어 있다.
- 사용자는 "LLM Wiki" 패턴을 개념적 기반으로 삼고 싶어 한다.
- 사용자의 핵심 문제는 세션 안에서 공부한 내용이 세션 밖에서는 사라진다는 점이다.
- 사용자는 즉시 구현이 아니라 먼저 `ralplan` 방식의 계획 수립을 원한다.
- 참고 아이디어 문서는 raw sources, wiki, schema/instructions의 3계층 구조를 강조한다.
- 같은 문서는 ingest, query, lint 워크플로와 `index.md`, `log.md`의 역할도 강조한다.

## 제약 조건

- 큰 플랫폼보다 작고 검토 가능한 MVP를 우선한다.
- 외부 API, 호스팅 DB, 임베딩 인프라가 반드시 필요하다고 가정하지 않는다.
- 시스템은 개인 워크플로에 잘 맞아야 하고, 결과물은 평범한 마크다운으로 읽기 쉬워야 한다.
- 초기 계획은 이후 더 풍부한 위키/블로그 형태로 발전 가능해야 하지만, 처음부터 과하게 넓게 잡지는 않는다.

## 미해결 질문 / 오픈 포인트

- 첫 ingest 입력은 수동 복붙, 내보낸 대화 기록, 직접 API 로깅 중 무엇을 우선할지
- 첫 인터페이스를 CLI 중심으로 둘지, 로컬 데스크톱 친화적으로 둘지, 웹으로 둘지
- 대화 내용을 위키에 반영하기 전 사용자 리뷰를 얼마나 거치게 할지
- "블로그 모드"를 자동 생성으로 둘지, 위키에서 필요할 때만 생성할지
- 대화 기반 주장에 어느 정도 세밀한 인용 단위를 둘지

## 예상 코드베이스 접점

- ingest 및 게시 흐름을 위한 신규 앱/부트스트랩 파일
- `raw/`, `wiki/`, `blog/`, `system/` 같은 위키 콘텐츠 디렉터리 구조
- 콘텐츠 규칙과 유지보수 워크플로를 정의하는 schema/instruction 파일
- ingest, query, lint 동작을 위한 로컬 스크립트/CLI 명령
