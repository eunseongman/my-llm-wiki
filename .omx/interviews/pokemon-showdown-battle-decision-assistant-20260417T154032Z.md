# Summary

- 초기 아이디어는 `선출 추천 + 턴별 추천 + 설명 + 메타 RAG`가 섞인 넓은 구상이었음.
- 인터뷰를 통해 본체가 `코칭 시스템`이 아니라 `설명 가능한 Pokemon Showdown 배틀 AI`임을 확인함.
- 평가 기준은 `학습 만족도`가 아니라 `규칙 기반 베이스라인보다 높은 승률`로 수렴함.
- 입력은 `Showdown 로그 파싱`, 정보 경계는 `공개 정보 + 시즌 메타 기반 확률 추정`으로 정리됨.
- MVP 비범위는 시즌/포맷 확장, 팀 빌딩, 복기/리플레이 해설, 자동 플레이/자동 입력으로 잠김.

# Pressure-pass findings

1. `설명 가능한 코칭 시스템`이라는 표현은 실제 목표를 가렸음.
   실제 성공 기준이 승률 중심이므로 프로젝트 본체는 의사결정 엔진 / 배틀 AI에 가깝다.

2. `입문자 대상`과 `비공개 정보까지 추정하는 최적 행동 추천`은 긴장 관계에 있다.
   입문자 도구로 포장하면 UX/신뢰 문제가 커지고, 실제로는 상당히 어려운 추론 엔진 문제다.

3. `RAG 프로젝트`라는 인상은 부정확했다.
   메타 정보는 추천의 재료이며, 핵심 난제는 상태 표현, 불완전 정보 처리, 행동 추천, 평가 루프다.

# Condensed transcript

## Round 1
- 사용자 정의, 문제 정의, 정체성, 입력 방식, 가치 기준을 압박 질문으로 시작.

## Round 2
- 사용자 답변은 입문자/상위권 유저를 동시에 타깃으로 잡고, 선출과 턴 의사결정을 둘 다 핵심 문제로 제시.
- 코칭 시스템이라면서 추천이 틀리면 가치가 거의 없다고 답해 정체성 충돌이 드러남.

## Round 3
- 입문자를 남기고도 선출 추천과 턴별 추천을 둘 다 고수.
- 비공개 정보 추정은 LLM, 출력은 추천+근거, 입력은 사실상 수동 혹은 로그 기반 가정.

## Round 4
- 사용자 압박 후 프로젝트의 본질이 배틀 AI라는 점을 인정.
- 평가 기준을 승률 기여로 이동시킴.

## Round 5-7
- 입력은 Showdown 로그 파싱, 평가 환경은 실제 래더, 성공 기준은 규칙 기반 베이스라인 초과로 정리.
- 비공개 정보는 시즌 메타 통계를 활용한 확률 추정으로 처리.

## Round 8
- MVP 비범위로 다음을 고정:
  - 특정 시즌/특정 포맷 외 지원 안 함
  - 팀 빌딩 추천 안 함
  - 복기/리플레이 해설 안 함
  - 자동 플레이/자동 입력 안 함
