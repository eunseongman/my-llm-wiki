## 초기 요청

RAG 팀 프로젝트 주제를 기존의 AI 기반 신기술 탐색 성격에서,
RAG 파이프라인과 문서 시스템을 잘 보여줄 수 있는 방향으로 바꾸고 싶다.

## 라운드 요약

### Round 1
- 질문: 새 주제를 고를 때 가장 우선하는 기준은 무엇인가?
- 답변: 공개 데이터로 빠르게 구현 가능하고, RAG 전 과정을 잘 보여주는 것이 1순위다.
- 해석: 평가 기준이 결과물 화려함보다 구현 가능성과 파이프라인 시연성에 맞춰져 있다.

### Round 2
- 질문: 단일 시험 심화형인지, 여러 자격증 플랫폼형인지?
- 답변: MVP는 정보처리기사 실기처럼 시험 하나만 깊게 가자.
- 해석: 범위를 좁혀서 데이터 품질과 검색/평가 설계를 더 선명하게 보여주려는 의도다.

### Round 3
- 질문: 이번 MVP에서 확실히 제외할 것은 무엇인가?
- 답변:
  - 여러 자격증 지원 제외
  - 개인 맞춤 학습 스케줄링 제외
  - 실시간 웹 크롤링 제외
  - 로그인/사용자 관리 제외
  - 문제 자동생성, 정답률 기반 추천도 1차 MVP에서는 제외
- 해석: 프로젝트를 학습 플랫폼이 아니라 문서 기반 RAG 실험/설계 프로젝트로 유지하려는 경계가 명확해졌다.

### Round 4
- 질문: 발표/시연에서 최소 성공으로 볼 핵심 기능 2개는?
- 답변:
  - 시험 범위 질문에 근거 문서 인용 답변
  - 특정 개념 요약 생성
  - 퀴즈/암기카드는 있으면 좋지만 후순위
- 해석: MVP 핵심 기능과 확장 기능의 구분이 생겼다.

### Round 5
- 질문: 발표용 초안에서 대상 시험, 데이터 종류, 확장 기능 위치를 고정해도 되는가?
- 답변: 일단 고정하자.
- 해석: 실행 가능한 제안문 형태로 정리해도 되는 의사결정 경계가 확보되었다.

## 압축된 결론

이번 프로젝트는 `정보처리기사 실기 한 과목에 집중한 RAG 학습 도우미`로 정의한다.
핵심은 서비스 확장이 아니라, 공개 문서를 수집하고 문서 유형별로 청킹하고,
임베딩/검색/리랭킹/평가를 설계하여 근거 기반 답변과 개념 요약을 제공하는 과정을
보여주는 것이다.

## Pressure Pass

- 처음에는 `자격증 취득 도우미`가 다자격증 지원이나 플랫폼형 서비스로 커질 가능성이 있었다.
- 이에 대해 단일 시험 심화형인지 확인했고, 이어서 비목표를 명시하게 하여 범위를 강제로 축소했다.
- 그 결과 "무엇을 만들까"보다 "무엇을 만들지 않을까"가 명확해졌고,
  MVP의 본질이 학습 플랫폼이 아니라 문서 기반 RAG 설계 데모라는 점이 분명해졌다.
